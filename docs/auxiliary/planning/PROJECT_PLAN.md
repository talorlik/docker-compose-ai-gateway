# Project Plan: Local AI Microservice Mesh

## 1. Project Overview

### 1.1 Goal

Build a locally runnable, multi-container microservice system where an AI classifier
selects the backend service for each request. The browser UI displays the exact
path taken (application-level tracing) per action, demonstrating Docker Compose
advanced features and AI-driven routing.

### 1.2 Demo Surface

The project must be demonstrable via:

- **Browser UI** (primary): Single-page application (Query, Train, Refine) showing
  routing decisions, request traces, and train/refine job results via SSE
- **Terminal commands**: curl-based API testing with trace visibility
- **Scaling scenarios**: Demonstrate load distribution across scaled services
- **Failure scenarios**: Show fallback behavior when backends fail

### 1.3 Technical Requirements

- **Platform**: Cross-platform (Linux, Mac, Windows)
- **Language**: Python (FastAPI for all services)
- **AI Model**: scikit-learn (TF-IDF + Logistic Regression) - see Section 5 for
details
- **Tracing**: Application-level tracing (no external observability stack) - see
Section 2.4
- **Docker**: Multi-stage Dockerfiles with pre-built model
  artifacts - see Section 8
- **Docker Compose**: Advanced features including fragments, anchors, profiles,
health checks - see Section 9

## 2. System Architecture

### 2.1 Services

1. **gateway** (FastAPI)
   - Serves static UI at `/` (Query | Train | Refine; Section 7.3)
   - Main API entrypoint: `POST /api/request`
   - Calls `ai-router` to classify requests; proxies to backends and
     training-api. See [ARCHITECTURE.md](docs/auxiliary/architecture/ARCHITECTURE.md).
   - Aggregates and returns full request trace
   - Implements confidence thresholding and fallback logic (see Section 2.3)

2. **ai-router** (FastAPI)
   - Endpoint: `POST /classify`
   - Runs lightweight text classifier (see Section 5 for model details)
   - Returns route, confidence score, and explanation (top weighted features)
   - Loads pre-built model artifact at startup

3. **search-service** (FastAPI)
   - Endpoint: `POST /handle`
   - Simulates information lookup/research responses

4. **image-service** (FastAPI)
   - Endpoint: `POST /handle`
   - Simulates image-related request handling

5. **ops-service** (FastAPI)
   - Endpoint: `POST /handle`
   - Simulates DevOps/infrastructure troubleshooting responses

6. **trainer** (one-shot container, no FastAPI)
   - Invoked by training-api or directly via profile. See Section 8.3,
     Section 9.3, and [ARCHITECTURE.md](docs/auxiliary/architecture/ARCHITECTURE.md).

7. **refiner** (one-shot container, no FastAPI)
   - Invoked by training-api or directly via profile; promotion via
     training-api or `scripts/promote.sh`. See
     [REFINER_PLAN.md](docs/auxiliary/refiner/REFINER_PLAN.md),
     [REFINER_TECHNICAL.md](docs/auxiliary/refiner/REFINER_TECHNICAL.md),
     [REFINER_FLOW.md](docs/auxiliary/refiner/REFINER_FLOW.md).

8. **training-api** (FastAPI)
   - Train, refine, promote (HTTP + CLI). See
     [TRAIN_AND_REFINE_GUI_PAGES_TECH.md](docs/auxiliary/architecture/TRAIN_AND_REFINE_GUI_PAGES_TECH.md),
     [ARCHITECTURE.md](docs/auxiliary/architecture/ARCHITECTURE.md).

9. **Redis**
   - Job state and Pub/Sub for training-api. See
     [TRAIN_AND_REFINE_GUI_PAGES_TECH.md](docs/auxiliary/architecture/TRAIN_AND_REFINE_GUI_PAGES_TECH.md),
     [ARCHITECTURE.md](docs/auxiliary/architecture/ARCHITECTURE.md).

### 2.2 Request Flow

1. User submits request via UI or curl → `gateway /api/request`
2. Gateway calls → `ai-router /classify` with request text
3. AI router returns → `{route, confidence, explanation, trace_append}`
4. Gateway evaluates confidence threshold:
   - If route is `unknown` → return 404-equivalent response
   - If confidence below threshold → treat as `unknown`
   - Otherwise → proxy to selected backend `/handle`
5. Backend returns → `{payload, trace_append}`
6. Gateway aggregates trace and returns → complete response with routing metadata

### 2.3 Routing Model

**Routes (4 labels):**

- `search-service`: Lookup, explanation, comparison queries
- `image-service`: Image-related intents (detection, processing, etc.)
- `ops-service`: DevOps, infrastructure, troubleshooting queries
- `unknown`: Out-of-scope, unclear, or irrelevant requests → returns 404-equivalent

**Routing Policy:**

- Model produces probabilities for all 4 labels
- Gateway uses confidence threshold (`T_route`, e.g., 0.55) for non-unknown routes
- Optional margin check: if difference between top two routes < threshold → treat
as unknown
- If route is `unknown` or confidence too low → return 404 without proxying to backend

### 2.4 Application-Level Tracing

#### 2.4.1 Trace Contract

- `request_id` (UUID) generated by gateway if not provided, propagated end-to-end
via headers and body
- Each service appends exactly one trace entry per request
- Gateway aggregates all entries into single `trace` array
- Trace returned to UI/client in gateway response

#### 2.4.2 Trace Entry Schema

```json
{
  "service": "gateway",
  "event": "received",
  "ts": "2026-02-11T19:12:01.210Z",
  "meta": {
    "detail": "...",
    "route": "ops-service",
    "confidence": 0.87,
    "status": 200
  }
}
```

#### 2.4.3 Gateway Response Schema

Trace entries follow the schema defined in Section 2.4.2.

```json
{
  "request_id": "c7a8...",
  "route": "ops-service",
  "confidence": 0.87,
  "explanation": "top tokens: kubectl, pod, crashloop",
  "trace": [
    {"service": "web", "event": "submit", "ts": "..."},
    {"service": "gateway", "event": "received", "ts": "..."},
    {"service": "ai-router", "event": "classified", "ts": "...", "meta": {"route": "ops-service", "confidence": 0.87}},
    {"service": "ops-service", "event": "handled", "ts": "...", "meta": {"status": 200}},
    {"service": "gateway", "event": "responded", "ts": "...", "meta": {"status": 200}}
  ],
  "backend_response": {
    "payload": {}
  },
  "timings_ms": {
    "classify": 12,
    "proxy": 18,
    "total": 35
  }
}
```

#### 2.4.4 Unknown Route Response (404-equivalent)

Trace entries follow the schema defined in Section 2.4.2. See Section 2.3 for
routing policy.

```json
{
  "request_id": "c7a8...",
  "route": "unknown",
  "confidence": 0.81,
  "message": "Unable to determine a suitable backend for this request",
  "trace": [
    {"service": "web", "event": "submit", "ts": "..."},
    {"service": "gateway", "event": "received", "ts": "..."},
    {"service": "ai-router", "event": "classified", "ts": "...", "meta": {"route": "unknown", "confidence": 0.81}},
    {"service": "gateway", "event": "responded", "ts": "...", "meta": {"status": 404}}
  ]
}
```

## 4. Training Data Strategy

### 4.1 Data Source: Synthetic, Hand-Written

**Primary approach:** Generate curated examples yourself.

**Why:**

- You control intent definitions
- Deterministic, explainable behavior
- Coverage of language patterns more important than realism
- No external dependencies

**Requirements:**

- **Minimum**: 50-100 examples per label
- **Recommended**: 150-300 examples per label
- **Distribution**: Balanced across all 4 labels (search, image, ops, unknown)

### 4.2 Data Generation Guidelines

**search-service examples:**

- Patterns: "what is...", "compare...", "explain...", "find...", "best way to..."
- Examples:
  - "compare nginx ingress and traefik"
  - "what is a reverse proxy"
  - "explain docker networking modes"
  - "find the latest stable nginx config pattern"

**image-service examples:**

- Patterns: image nouns, vision verbs, file references
- Examples:
  - "detect objects in an image"
  - "blur faces in a photo"
  - "generate captions for pictures"
  - "this is about an image: detect objects in a photo"

**ops-service examples:**

- Patterns: tooling names, error words, imperative debugging tone
- Examples:
  - "kubectl pods stuck in CrashLoopBackOff"
  - "terraform apply fails with provider error"
  - "docker container exits immediately debug"
  - "kubernetes pod can't pull image, debug steps"

**unknown examples:**

- Patterns: chit-chat, vague questions, nonsense, generic conversation
- Examples:
  - "hello"
  - "tell me a joke"
  - "what do you think about life"
  - "asdf qwer"
  - "what is love"

### 4.3 Optional Augmentation

**Stack Overflow titles (for ops):**

- Source: Stack Overflow dumps or Kaggle "StackLite"
- Filter by tags: docker, kubernetes, terraform, aws
- Label as `ops`

**Intent datasets (for general patterns):**

- CLINC / OOS-style datasets
- Map multiple intents into search/unknown categories

**Note:** External data is optional and additive. Synthetic data is sufficient
for this project.

## 5. AI Model Implementation

### 5.1 Model Choice

**Algorithm:**

- `TfidfVectorizer(ngram_range=(1,2), min_df=2)`
- `LogisticRegression(max_iter=2000, solver="lbfgs", multi_class="multinomial")`

**Rationale:**

- Fast inference on CPU (no GPU required)
- Deterministic behavior (reproducible results)
- Small model artifact size
- Provides probability outputs (`predict_proba`) for confidence scoring
- Explainable via feature weights

### 5.2 Model Characteristics

This is **real AI** (machine learning-based classification):

- Trained on data, not rule-based
- Produces probabilistic inference
- Behavior changes with training data
- Performs inference on unseen inputs

It is **not**:

- A generative language model (LLM)
- Deep learning or neural networks
- Capable of text generation or reasoning

### 5.3 Training Pipeline

**Training code and data:** `services/trainer/`

- `train.py`: Training script
- `train.csv`: Training data (`text,label` format)

**Model artifact output:** `services/ai_router/model/`

- `model.joblib`: Pre-built model artifact (produced by the
  trainer service, or via shared volume at runtime)

**Training Process:**

1. Load `train.csv` with fixed `random_state` for reproducibility
2. Split train/validation sets
3. Fit `TfidfVectorizer` on training text
4. Fit `LogisticRegression` on vectorized features
5. Evaluate accuracy and confusion matrix
6. Save artifact containing:
   - Vectorizer
   - Model
   - Label mapping

**Training Data Requirements:**

- See Section 4 for detailed training data strategy and requirements

### 5.4 Explainability

Generate explanation in inference:

- Extract top weighted features for predicted class
- Return 3-8 most significant tokens/phrases
- Display in UI as "explanation" field

### 5.5 Inference Behavior

1. Load `model.joblib` at service startup
2. Vectorize input text using trained vectorizer
3. Compute probabilities: `probs = model.predict_proba(X)`
4. Select route: `route = argmax(probs)`
5. Compute confidence: `confidence = max(probs)`
6. Generate explanation from top features
7. Return `{route, confidence, explanation, trace_append}`

## 8. Docker Implementation

### 8.1 Multi-Stage Dockerfile Pattern (Python Services)

**Standard pattern for gateway and backend services:**

```dockerfile
# Builder stage
ARG PY_IMAGE=python:3.12-slim-bookworm

# Stage 1: Build wheels
FROM ${PY_IMAGE} AS builder
WORKDIR /build
COPY requirements.txt .

# Build wheels for faster installs and better caching
RUN pip wheel --no-cache-dir -r requirements.txt -w /wheels

# Runtime stage
FROM ${PY_IMAGE} AS runtime
ENV DEBIAN_FRONTEND=noninteractive

# Create non-root user for security
RUN groupadd -r appgroup \
    && useradd -r -g appgroup -m -d /home/appuser -s /usr/sbin/nologin appuser

WORKDIR /app

# Install from pre-built wheels (fast, no compilation needed)
COPY --from=builder /wheels /wheels
COPY requirements.txt .
RUN pip install --no-cache-dir --no-index --find-links=/wheels -r requirements.txt \
    && rm -rf /wheels  # Clean up wheels after install

# Python optimizations
ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

# Copy application code
COPY --chown=appuser:appgroup app/ ./app/

USER appuser
EXPOSE 8000

# Healthcheck for production monitoring
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD python -c "import sys,urllib.request; urllib.request.urlopen('http://127.0.0.1:8000/health', timeout=2); sys.exit(0)"

# Use gunicorn with uvicorn workers for production
CMD ["gunicorn", "app.main:app", \
    "--bind", "0.0.0.0:8000", \
    "--workers", "2", \
    "--worker-class", "uvicorn.workers.UvicornWorker", \
    "--access-logfile", "-", \
    "--error-logfile", "-"]
```

**Dev target (optional):**

```dockerfile
FROM runtime AS dev
# Override CMD for development with hot reload
# Note: uvicorn --reload requires root or proper file permissions
# For dev, we can temporarily allow this or use a different approach
USER root
RUN chown -R appuser:appgroup /app
USER appuser
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000", "--reload"]
```

### 8.3 Trainer Dockerfile

**Design:** Single-stage image with training dependencies only (no
FastAPI, no gunicorn). `train.py` is COPYed at build time; `train.csv`
is mounted from the host at runtime. The trainer writes artifacts
(`model.joblib`,
`metrics.json`, `misclassified.csv`) into a named volume shared
with ai-router. See Section 9.3 for the Compose configuration.

```dockerfile
ARG PY_IMAGE=python:3.12-slim-bookworm
FROM ${PY_IMAGE}
WORKDIR /train

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY train.py .

ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1

CMD ["python", "train.py", \
    "--data", "./train.csv", \
    "--out", "/model/model.joblib", \
    "--metrics", "/model/metrics.json", \
    "--misclassified", "/model/misclassified.csv"]
```

**`requirements.txt` (training deps only):**

```text
scikit-learn==1.5.2
numpy==2.1.3
joblib==1.4.2
```

### 8.4 Dockerfile Dependencies

**Required Python packages for production:**

- `gunicorn` - Production WSGI server (required for production CMD)
- `uvicorn[standard]` - ASGI server (used as gunicorn worker class)
- Other application dependencies as specified in `requirements.txt`

**Note:** Ensure `gunicorn` is included in `requirements.txt` for all services
using the production Dockerfile pattern.

### 9.3 Trainer Service and Shared Volume

**Named volume:**

A `model_artifacts` volume is shared between the trainer (read-write) and
ai-router (read-only). The trainer writes `model.joblib`, `metrics.json`,
and `misclassified.csv` into the volume; ai-router reads `model.joblib`
at startup via `MODEL_PATH`.

**Compose definition:**

```yaml
volumes:
  model_artifacts:

services:
  trainer:
    build:
      context: ../services/trainer
      dockerfile: Dockerfile
    profiles:
      - train
    volumes:
      - ../services/trainer/train.csv:/train/train.csv:ro
      - model_artifacts:/model
    environment:
      <<: *common-env
    logging:
      <<: *common-logging
```

**ai-router volume mount:**

```yaml
  ai_router:
    volumes:
      - model_artifacts:/model:ro
    environment:
      MODEL_PATH: /model/model.joblib
```

**Workflow:**

1. Edit `services/trainer/train.csv` on the host.
2. Run the trainer (no rebuild needed):
   `docker compose --profile train run --rm trainer`.
3. Optionally run the refiner: `docker compose --profile refine run --rm refiner`;
   then `./scripts/promote.sh` to promote only if metrics improve.
4. Restart ai-router: `docker compose restart ai_router`.
5. Rebuild the trainer image only when `train.py` or `requirements.txt`
   change; `train.csv` is mounted at runtime.

**Model loading in ai-router:** At startup, load from `MODEL_PATH` env
var if set and the file exists; otherwise fall back to the build-time
artifact at `/app/model/model.joblib`. Fail fast if neither exists.

**Optional hot-reload:** Add `POST /reload-model` to ai-router for
model swap without restart (thread-safe lock, internal admin only). This
is not required; restart is the simplest correct approach.

**Steps:**

2. Send image-like prompt: `curl -X POST ... -d '{"text": "detect objects in

3. Observe gateway behavior

**Expected Behavior:**

- AI router classifies as image-service
- Gateway attempts to proxy to image-service
- Proxy fails (service unavailable)
- Gateway returns error response with trace showing:
  - Classification to image-service
  - Backend failure event
  - No fallback routing (service is down, not low confidence)

**Note:** This demonstrates infrastructure failure, distinct from `unknown` classification.

### 10.5 Unknown vs Failure Distinction

**Unknown (AI Decision)** - see Section 2.3 (Routing Policy) and Section 2.4.4
(Unknown Route Response):

- Trace shows: `ai-router` classified as `unknown`
- No backend hop occurs
- HTTP 404 response
- Message: "Unable to determine a suitable backend for this request"

**Backend Failure (Infrastructure):**

- Trace shows: `ai-router` classified to specific route
- Backend hop attempted but failed
- Error in trace metadata
- Different error message indicating service unavailability

## 13. Technical Notes

### 13.1 Why This Approach

**AI as Control Plane:**

- Demonstrates AI making routing decisions, not just processing content
- Shows ML inference as an internal platform capability
- Proves Docker Compose can model complex system behavior

**Application-Level Tracing:**

- No external observability stack required
- Deterministic, user-visible path per request
- Replaces distributed tracing UIs with embedded visualization

**Classical ML vs LLM** (see Section 5.2 for model characteristics):

- Fast, deterministic, explainable
- Runs on CPU without external dependencies
- Small artifact size, easy to ship in containers
- Sufficient for intent classification use case

### 13.2 Key Design Decisions

1. **Python-only**: Simplifies development and maintenance
2. **UI served from gateway**: Reduces container count
3. **Pre-built model artifact**: Deterministic, reproducible
   models produced by the trainer service and copied into
   ai-router at build time
4. **Self-contained trainer service**: One-shot container with
   its own training code and data; writes artifacts to a shared
   volume for ai-router to consume
5. **4-route model**: Explicit unknown handling vs implicit fallback
6. **Confidence thresholding**: Prevents forced misrouting
7. **Application tracing**: No observability stack dependency

### 13.3 Refiner and Dataset Improvement

The refiner service automates dataset improvement using a local LLM. See
[REFINER_PLAN.md](docs/auxiliary/refiner/REFINER_PLAN.md),
[REFINER_TECHNICAL.md](docs/auxiliary/refiner/REFINER_TECHNICAL.md),
[REFINER_FLOW.md](docs/auxiliary/refiner/REFINER_FLOW.md).

### 13.4 Centralized Configuration

Configuration for Redis, Ollama, refine parameters, and key paths is defined
once in `config/PROJECT_CONFIG.yaml` and materialized into `env/.env.<env>`
files via `scripts/generate_env.py`. Docker Compose consumes these env files
with `env_file`, and services continue to read values via environment
variables. For details, see `CONFIGURATION.md`.

### 13.5 Future Enhancements (Optional)

- Evaluator and promoter services (beyond scripts/promote.sh)
- Enhanced explanations (top n-grams with weights)
- Request replay endpoint for debugging
- Metrics endpoint for gateway (request rate, latency per route)
- Additional backend services
- More sophisticated load balancing in gateway
- On-demand Ollama model selection: UI dropdown to list, choose, or pull
  LLM models for the refiner (via Ollama `/api/tags` and `/api/pull`),
  with download progress feedback in the UI
- Reduce `AUGMENT_MIN_PER_LABEL` further (e.g., from 5 to 3) to cut
  augmentation Ollama calls and speed up refinement
- Two-phase refinement: split into a fast "relabel-only" pass (< 2 min)
  that returns immediately, and a separate "augment" pass that runs
  asynchronously and streams results as they arrive
